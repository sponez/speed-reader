# Speed Reader: целевая архитектура

## Подход

Для первой версии подходит лёгкая layered / ports-and-adapters архитектура без тяжёлой clean architecture церемонии.

Главная цель архитектуры - отделить доменную механику чтения от React, Tauri и конкретного способа отрисовки. Это позволит тестировать токенизацию, расчёт скорости и окно видимости обычными unit-тестами, а UI оставить адаптером над этой логикой.

## Слои

### Tauri shell

Отвечает за desktop-окружение:

- создание Windows-приложения;
- сборку installer-артефакта;
- подключение Tauri Store;
- минимальные системные возможности.

Rust-слой должен оставаться тонким. В MVP бизнес-логика чтения не переносится в Rust.

Текущий scaffold:

- `src-tauri/tauri.conf.json` задаёт `productName` и title окна как `Speed Reader`;
- `identifier` установлен в `com.local.speedreader`;
- frontend dev URL: `http://localhost:1420`;
- production frontend берётся из `../dist`.

### React app

Отвечает за пользовательский сценарий:

- экран подготовки;
- запуск попытки;
- режим чтения;
- обработку `Esc`;
- передачу настроек в доменный слой;
- отображение состояния чтения.

React-компоненты не должны самостоятельно считать WPM, токены или границы окна видимости. Они вызывают функции доменного слоя и рендерят результат.

Текущий React/Vite scaffold находится в `src`. На следующем шаге его нужно разложить по архитектурным границам, отделив доменную логику чтения от компонентов и стилей.

### Reading domain

Чистый TypeScript-слой без зависимости от React, DOM и Tauri.

Отвечает за:

- нормализацию входного текста;
- токенизацию на слова и разделители;
- расчёт длительности одного слова по WPM;
- расчёт текущего слова по прошедшему времени;
- расчёт границ окна видимости;
- завершение сессии при достижении конца текста.

Этот слой должен быть главным местом unit-тестов.

### Rendering adapter

Отвечает за связь доменной модели с DOM-отрисовкой:

- отображает токены текста;
- измеряет или использует DOM-позиции слов, если они нужны для плавного движения;
- применяет CSS-классы для активного слова, видимого окна и размытого текста;
- обеспечивает визуальную стабильность при длинном тексте и переносах строк.

Rendering adapter может зависеть от React и DOM, но не должен менять доменную логику.

### Persistence adapter

Отвечает за локальное сохранение:

- читает настройки и последний текст из Tauri Store;
- сохраняет изменения перед запуском или при изменении настроек;
- предоставляет React app простой интерфейс загрузки и сохранения.

Доменный слой не знает о Tauri Store.

Tauri Store ещё не подключён в scaffold. Подключение плагина и адаптера хранения остаётся отдельной задачей MVP.

## Концептуальные типы

```ts
type ReaderSettings = {
  wpm: number;
  visibleWordsBefore: number;
  visibleWordsAfter: number;
  blurIntensity: number;
};

type ReadingText = {
  rawText: string;
  tokens: Token[];
};

type Token = {
  id: string;
  text: string;
  kind: "word" | "separator";
  wordIndex?: number;
};

type ReadingSession = {
  text: ReadingText;
  settings: ReaderSettings;
  startedAt: number;
  status: "idle" | "running" | "finished";
};

type FocusWindow = {
  activeWordIndex: number;
  firstVisibleWordIndex: number;
  lastVisibleWordIndex: number;
};

type ReadingProgress = {
  activeWordIndex: number;
  elapsedMs: number;
  isFinished: boolean;
};
```

Имена и точная форма типов могут измениться при реализации, но эти сущности должны остаться разделёнными по смыслу.

## Поток данных

1. Пользователь вводит текст и настройки на экране подготовки.
2. React app передаёт текст в reading domain.
3. Reading domain создаёт `ReadingText` с токенами слов и разделителей.
4. При запуске создаётся `ReadingSession`.
5. Rendering adapter отображает токены в DOM.
6. После старта таймер или `requestAnimationFrame` передаёт elapsed time в reading domain.
7. Reading domain рассчитывает `ReadingProgress` и `FocusWindow`.
8. React app передаёт эти данные rendering adapter.
9. Rendering adapter обновляет подсветку, окно видимости и blur.
10. Нажатие `Esc` завершает сессию и возвращает приложение к экрану подготовки.

## Правила зависимостей

- Reading domain не импортирует React, DOM API, Tauri API и CSS.
- React app может импортировать reading domain, rendering adapter и persistence adapter.
- Persistence adapter не содержит логики WPM, токенизации или окна видимости.
- Rendering adapter не сохраняет данные и не управляет настройками.
- Tauri shell не содержит пользовательский сценарий чтения.

## Состояния приложения

Минимальные состояния MVP:

- `preparing` - пользователь вводит текст и меняет настройки.
- `reading` - попытка запущена, настройки заблокированы, паузы нет.
- `finished` - текст закончился или пользователь нажал `Esc`; приложение возвращается к подготовке.

Для MVP можно не держать отдельный экран завершения. После завершения попытки достаточно вернуться к `preparing`.

## Обработка ошибок и ограничений

- Пустой текст не запускает чтение.
- Некорректный WPM приводится к допустимому диапазону на уровне UI/настроек.
- Очень длинный текст должен оставаться читаемым, но виртуализация не обязательна для MVP.
- Если Tauri Store недоступен, приложение должно продолжить работу с настройками по умолчанию.

## Тестирование архитектуры

Unit-тесты:

- токенизация пустого текста;
- токенизация текста с пробелами и переносами;
- токенизация пунктуации;
- расчёт длительности слова по WPM;
- расчёт активного слова по elapsed time;
- расчёт окна видимости на начале, середине и конце текста.

UI/smoke-тесты:

- загрузка экрана подготовки;
- сохранение и восстановление настроек;
- запуск чтения скрывает подготовительный UI;
- `Esc` завершает попытку;
- активное слово имеет визуальный класс подсветки;
- слова вне окна видимости имеют визуальный класс размытия.

## Решения для MVP

- Скорость измеряется в WPM, не в символах в минуту и не в пикселях в секунду.
- Окно видимости измеряется количеством слов вокруг текущего слова.
- Во время чтения нельзя менять настройки.
- Пауза не реализуется.
- История сессий и статистика не реализуются.
- Первый production artifact - Windows installer через Tauri build.
